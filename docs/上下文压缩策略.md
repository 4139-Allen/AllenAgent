## 整体流程

三个阶段按顺序协同工作，每条工具结果都经过完整的三层处理：

```
工具原始输出
    │
    ▼
┌─────────────────────────────────────────────────┐
│ ① 生命周期判断（utils/tool_lifecycle.py）         │
│    ├── ephemeral → 轻量截断 → 存入 memory         │
│    │                   （LLM 下一轮看不到）         │
│    └── persistent → 继续                          │
└─────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────┐
│ ② 需要摘要？（utils/tool_summarizer.py）           │
│    ├── True → 非流式 LLM 摘要 → 存入 memory       │
│    │            （不截断，摘要已足够短）             │
│    └── False → 继续                               │
└─────────────────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────┐
│ ③ 按工具类型截断（utils/tool_truncation.py）      │
│    5 种策略 + 语义过滤 + 硬性上限 2000 字符        │
│    ↓                                              │
│    存入 memory                                     │
└─────────────────────────────────────────────────┘
```

**三条原则：**
- 生命周期判断在最先（需要原始内容判断错误/异常）
- 摘要和截断互斥，对同一结果只选一种
- JSONL 保存完整轨迹（含 `_lifecycle` 标记），重新加载后过滤逻辑一致

## 第一阶段：工具结果生命周期

**实现文件：** `utils/tool_lifecycle.py`  
**修改文件：** `memory/short_term.py`、`agents/react_loop.py`、`agents/allen_agent.py`、`memory/conversation_store.py`

### 生命周期层级

| 层级 | 行为 | 示例 |
|-----|------|------|
| `persistent` | 保留在上下文中，受滑动窗口管理 | 文件内容、错误信息、搜索结果 |
| `ephemeral` | `get_messages()` 时自动过滤，LLM 看不到 | 目录列表、存在性检查 |
| `conditional` | 运行时根据内容决定（含错误 → persistent，否则看长度） | shell 执行 |

### 生命周期定义表

#### ephemeral（用完即丢）

| 工具 | 理由 |
|-----|------|
| `file_io(action=list)` | 目录列表，结果已展示在 UI，LLM 已读到 |
| `file_io(action=exists)` | 存在性检查，结论已体现在下一步行动里 |
| `file_io(action=write)` | 写操作结果已体现在 diff 中 |

#### persistent（必须保留）

| 工具 | 理由 |
|-----|------|
| `file_io(action=read)` | 文件内容，后续修改需要对照 |
| `file_io(action=delete)` | 删除结果，后续可能需要回滚 |
| `search_web` | 事实依据，不能丢 |
| `search_knowledge_base` | RAG 检索结果 |

#### conditional（运行时判断）

| 工具 | 判断逻辑 |
|-----|---------|
| `shell` / `execute_command` | 含错误/异常 → persistent；内容 < 500 字符 → persistent；否则 ephemeral |

### 判断逻辑

`decide_lifecycle(tool_name, content, action=None)` 查找顺序：

1. `tool_name(action=action)` — 如 `"file_io(action=read)"`
2. 纯工具名 — 如 `"shell"`
3. 默认 `"persistent"`

conditional 的判断：
- 含 error/traceback/exception/失败/拒绝/错误/异常 → persistent
- 短输出（< 500 chars）→ persistent
- 长输出（>= 500 chars）→ ephemeral

### 读取时过滤

`get_messages()` 调用 `_filter_ephemeral()`：

- **纯 ephemeral 批**（某批 `assistant(tool_calls)` 所有结果都是 ephemeral）→ 整批跳过
- **混合批**（部分 ephemeral 部分 persistent）→ 整批保留，不破坏消息配对
- `_lifecycle` 标记从返回结果中剥离，不暴露给 LLM

### JSONL 持久化

- `get_history()` 保留 `_lifecycle` 标记 → JSONL 写入时序列化为 `lifecycle` 字段
- JSONL 加载时恢复 `_lifecycle` → 重新加载后 ephemeral 过滤仍然生效

## 第二阶段：工具输出截断

**实现文件：** `utils/tool_truncation.py`  
**修改文件：** `memory/short_term.py`

### 三层处理

```
原始内容 → ① 语义过滤 → ② 按策略截断 → ③ 2000 字符兜底
```

### 语义过滤

在截断前去除明显无意义的行。保守策略——过滤后剩余行数不足原 50% 时放弃过滤：

| 丢弃 | 保留 |
|------|------|
| `30%` 进度条 | 空行（段落分隔） |
| `[INFO]` 日志 | 错误/异常/警告 |
| `耗时：0.5s` | |
| `----` 分隔线 | |
| `Time elapsed:` | |

### 按工具类型截断策略

| 工具 | 策略 | 行为 |
|------|------|------|
| `file_io(action=read)` | 保留头部 | 文件路径标记 + 前 N 行代码结构 |
| `file_io(action=list)` | 保留尾部 | 目录头 + 尾部最新条目 |
| `shell`（含 error） | 错误上下文 | 保留最后一个错误行 ±10 行上下文 |
| `shell`（正常） | 保留尾部 | 结果在命令执行末尾 |
| `search_web` | 不截断 | 已限制 5 条 |
| `search_knowledge_base` | 不截断 | 同上 |
| 未知工具 | 保留头+尾 | 头部 1/3 + 尾部 2/3 |

### 写入时 vs 读取时

| 时机 | 方法 | 说明 |
|------|------|------|
| 写入时（`add_tool_result`） | `truncate_tool_output(name, content, action=...)` | 工具特定策略 |
| 读取时（`get_messages` 兜底） | `truncate_general(content)` | JSONL 加载的历史数据，通用头+尾截断 |

## 第三阶段：结果摘要化

**实现文件：** `utils/tool_summarizer.py`  
**修改文件：** `agents/react_loop.py`、`agents/allen_agent.py`

### 判断规则

`should_summarize(tool_name, content, action=None)`：

1. 长度 < **1500** 字符 → 否（不值得调一次 API）
2. 含错误/异常关键字 → 否（必须原样保留）
3. `search_web` / `search_knowledge_base` / `update_memory` → 否（已够紧凑）
4. `file_io`（所有 action）→ 否（截断优于摘要）
5. `code_search` → 否（grep/glob 结构化格式）
6. `shell` → 是（测试报告/数据分析结果需要提炼）
7. 兜底：未知工具长内容 → 是

### LLM 调用

```python
# 非流式、不用工具、单次请求、30s 超时
response = llm_provider.chat(
    messages=[{"role": "user", "content": prompt}],
    temperature=0.1, timeout=30.0, max_tokens=500,
)
```

- **不流式**：避免 DeepSeek V4 Flash `reasoning_content` 问题
- **不用 tool_calls**：纯文本请求，兼容所有提供商
- **失败降级**：返回 None → 调用方走截断
- **安全上限**：摘要结果超过 2000 字符时截断（与截断统一上限）
- **摘要内容不走截断策略**：截断对结构化的摘要没有意义，只保留 2000 字符安全兜底

### 摘要提示词

```
请用 2-3 句话提炼以下工具执行结果的核心结论。

要求：
- 只保留最关键的信息（最终结果、统计数字、关键数据、错误原因）
- 丢弃进度、重复内容、日志细节、中间状态
- 用中文回复
- 不超过 200 字
```


### 关键决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| ephemeral 过滤时机 | 读取时过滤（`get_messages`） | 保留原始数据用于 debug，JSONL 持久化保留完整轨迹 |
| 摘要模型 | 同供应商，非流式单次调用 | 避免 DeepSeek `reasoning_content` 问题；不需要额外 API key |
| 截断阈值 | 按工具策略 + 2000 统一兜底 | 兼顾灵活性和安全性 |
| 生命周期定义位置 | 独立模块 `utils/tool_lifecycle.py` | 与工具定义解耦，便于调整 |
| 摘要与截断关系 | 互斥，对同一结果只选一种 | 摘要已足够短，不需要再截断 |
| 语义过滤 | 保守策略，过滤 < 50% 原行数时放弃 | 避免误删有意义的行 |
