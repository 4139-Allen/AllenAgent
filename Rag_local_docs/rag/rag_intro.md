# RAG 检索增强生成

## 什么是 RAG

RAG（Retrieval-Augmented Generation，检索增强生成）是一种结合**向量检索**和**大语言模型**的技术。它通过在生成答案前先检索相关文档，显著提升 LLM 回答的准确性和可靠性。

## 为什么需要 RAG

### LLM 的局限性

1. **知识截止**：训练数据有时间限制，无法获取最新信息
2. **幻觉问题**：会编造看似合理但错误的内容
3. **私有知识缺失**：无法访问企业内部文档、个人笔记等

### RAG 如何解决

| 问题 | RAG 解决方案 |
|------|-------------|
| 知识过时 | 检索实时更新的文档 |
| 幻觉 | 基于真实文档生成，可追溯来源 |
| 私有知识 | 检索私有知识库 |

## RAG 工作流程

### 离线阶段：建库

```
原始文档 → 分块(Chunking) → 向量化(Embedding) → 存入向量数据库
```

1. **文档加载**：读取各种格式的文档（PDF、Word、Markdown等）
2. **文本分块**：将长文档切分为合适大小的片段
3. **向量化**：使用 Embedding 模型将文本转为向量
4. **存储索引**：存入向量数据库（ChromaDB、Qdrant、Pinecone等）

### 在线阶段：检索+生成

```
用户提问 → 向量化 → 检索相似文档 → 拼装Prompt → LLM生成答案
```

1. **Query 向量化**：将用户问题转为向量
2. **相似度检索**：从向量库中找到最相关的文档片段
3. **上下文拼装**：将检索结果作为参考资料
4. **LLM 生成**：基于参考资料生成最终答案

## 核心组件

### 1. 文档加载器（Document Loader）

支持多种格式：
- 纯文本：`.txt`, `.md`
- 文档：`.pdf`, `.docx`, `.pptx`
- 网页：`.html`
- 数据：`.csv`, `.json`

### 2. 文本分块器（Text Splitter）

常用策略：

| 策略 | 优点 | 缺点 |
|------|------|------|
| 固定长度 | 简单高效 | 可能切断语义 |
| 按句子 | 语义完整 | 块大小不均 |
| 按段落 | 保留结构 | 长段落需再分 |
| 递归分割 | 灵活适应 | 实现复杂 |

**推荐参数**：
- chunk_size: 300-500 字符
- overlap: 50-100 字符（保持上下文连续性）

### 3. Embedding 模型

将文本转为高维向量：

| 模型 | 维度 | 特点 |
|------|------|------|
| OpenAI text-embedding-3-small | 1536 | 效果好，需API |
| BAAI/bge-small-zh-v1.5 | 512 | 中文优化，免费 |
| sentence-transformers/all-MiniLM-L6-v2 | 384 | 轻量级，多语言 |

### 4. 向量数据库

| 数据库 | 特点 | 适用场景 |
|--------|------|----------|
| ChromaDB | 轻量级，嵌入式 | 本地开发、小规模 |
| Qdrant | 高性能，Rust实现 | 中大规模生产 |
| Pinecone | 全托管云服务 | 无需运维 |
| Milvus | 开源，可扩展 | 大规模部署 |

### 5. 检索策略

- **向量检索**：语义相似度匹配
- **关键词检索**：BM25 等传统方法
- **混合检索**：结合两者，效果最佳

## 关键参数调优

### 相似度阈值（score_threshold）

```
score > 0.7  → 高度相关
score > 0.5  → 相关
score > 0.3  → 低相关
score < 0.3  → 不相关（建议过滤）
```

### Top-K 检索数量

- 太少（1-2）：可能遗漏关键信息
- 适中（3-5）：平衡全面性和精确性
- 太多（>10）：引入噪音，增加成本

## 高级优化技巧

### 1. 查询改写（Query Rewriting）

```
原始问题："Python的GIL是什么？"
改写后："Python全局解释器锁 Global Interpreter Lock 工作原理"
```

### 2. 假设文档嵌入（HyDE）

先让 LLM 生成假设答案，用假设答案检索，找到更相关的真实文档。

### 3. 多轮检索

第一次检索结果不够时，基于结果进行二次检索。

### 4. 重排序（Reranking）

对初步检索结果用更精细的模型重新排序。

## 最佳实践

1. **分块大小**：根据文档类型调整，技术文档可小些，叙事文档可大些
2. **重叠设置**：保持 10-20% 的重叠率
3. **元数据**：保留文档来源、页码等信息，便于溯源
4. **定期更新**：知识库需要定期更新维护
5. **评估指标**：关注检索召回率和答案准确率
